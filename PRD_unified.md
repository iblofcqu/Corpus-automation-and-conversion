# 文档解析与结构化抽取算法统一 PRD

## 1. 项目背景

建筑安全领域的业务文档来源复杂，主要包括法律法规、事故调查报告和安全隐患图集。原始文件通常为 Word、PDF 或经 MinerU 解析后的半结构化结果，直接用于知识库、检索、问答或样本库建设时存在以下问题：

- 原始格式不统一，Word、PDF、图片和表格混杂。
- MinerU 可生成 Markdown、`content_list.json` 和图片资源，但标题层级、章节边界和业务字段仍需二次处理。
- 不同文档类型的结构差异明显，不能用一套固定规则覆盖所有任务。
- 业务系统历史对接的算法输出以 JSON 为主，但当前还需要明确提供“文件 -> Markdown”的标准化示例，作为后续抽取链路的统一输入说明。

本 PRD 将三个既有算法统一描述为一套“文档解析 + Markdown 标准化 + 类型化抽取”的能力体系。

## 2. 产品目标

构建一套面向建筑安全文档的统一算法能力，支持将不同类型的原始资料转换为可用于知识库建设、检索问答、样本整理和前端展示的结构化结果。

核心目标如下：

- 提供统一的输入输出说明，明确文件、Markdown、JSON、图片资源之间的关系。
- 说明历史对接形态：既有算法主要导出 JSON 结构化结果。
- 增加“文件 -> Markdown”的示例，说明当前链路中 Markdown 作为标准中间态的作用。
- 明确不同算法适用的任务类型：法律法规对应 A1，事故报告对应 C1，隐患图集提取对应 C3。
- 保留不同文档类型的专用处理逻辑，避免把法规、事故报告和图集强行合并成同一种抽取方式。

## 3. 任务类型映射

| 任务类型 | 文档类别 | 对应算法 | 默认输出 | 适用说明 |
|---|---|---|---|---|
| A1 | 法律法规 | 法律法规结构恢复算法 | `document.md` + `content_list.json` | 适用于法规、规范、标准、办法、条例等具有章、节、条结构的文本。 |
| C1 | 事故报告 | 事故调查报告本体抽取算法 | `*_ontology.json` + `*_memory.json` | 适用于事故调查报告、政府批复、调查组报告等需要按专家本体抽取字段的文档。 |
| C3 | 隐患图集提取 | 安全隐患图集提取算法 | 目录树 + `section_info.txt` + 图片 | 适用于安全隐患图集、图文样本库、隐患条目和配图对应关系抽取。 |

## 4. 历史对接说明：既有算法导出 JSON

历史对接中，算法侧主要向外部系统提供 JSON 结果，用于后续入库、检索或知识图谱构建。

典型输出包括：

- 法律法规 A1：导出重新标注标题层级后的 `content_list.json`，并同步生成 `document.md`。
- 事故报告 C1：导出本体约束后的 `*_ontology.json`，同时输出 `*_memory.json` 作为调试和溯源快照。
- 隐患图集 C3：当前主输出为目录树、图片和 `section_info.txt`；如需对接统一接口，可进一步封装为 JSON 清单，记录隐患路径、字段描述和图片路径。

因此，现有算法不是简单的 OCR 或文件转换，而是面向业务结构的 JSON/结构化结果导出能力。当前新增的“文件 -> Markdown”示例用于补充说明：在进入结构化抽取前，文件需要先转换为稳定的 Markdown 中间表示。

## 5. 文件 -> Markdown 示例

### 5.1 示例目标

将原始 Word/PDF 文件解析为标准化 Markdown，保留标题层级、正文、表格和图片引用，为后续 A1/C1/C3 算法提供统一输入。

### 5.2 输入示例

```text
input/
└── 建筑施工安全生产管理条例.pdf
```

或：

```text
input/
└── 某建设工程事故调查报告.docx
```

### 5.3 处理流程示例

```text
原始文件（Word/PDF）
        ↓
格式归一化
        - Word 转 PDF
        - PDF 直接进入解析
        ↓
MinerU 文档解析
        - 生成 Markdown
        - 生成 content_list.json
        - 提取 images/
        ↓
标题层级修正或结构恢复
        ↓
标准化 Markdown
```

### 5.4 输出示例

```text
output/建筑施工安全生产管理条例/
├── document.md
├── content_list.json
└── images/
    ├── page_1_img_1.jpg
    └── page_3_table_1.jpg
```

`document.md` 示例：

```markdown
# 建筑施工安全生产管理条例

## 第一章 总则

### 第一条

为了加强建筑施工安全生产管理，防止和减少生产安全事故，保障人民群众生命和财产安全，制定本条例。

### 第二条

在中华人民共和国境内从事建筑施工活动及其安全生产监督管理，应当遵守本条例。

## 第二章 安全生产责任

### 第三条

建设单位、施工单位、监理单位应当依法履行安全生产责任。
```

### 5.5 Markdown 的作用

Markdown 是结构化抽取前的标准中间态，不等同于最终业务结果。

- 对 A1 法律法规，Markdown 用于承载恢复后的章、节、条层级。
- 对 C1 事故报告，Markdown 用于给本体 Agent 提供稳定章节边界和正文上下文。
- 对 C3 隐患图集，Markdown 可作为辅助文本来源，核心仍以 `content_list.json` 中的标题、图片和表格节点为准。

## 6. 总体算法架构

```text
原始文件
  - Word
  - PDF
  - MinerU 解析结果
        ↓
文档解析与标准化
  - Word/PDF 归一化
  - MinerU 解析
  - Markdown / content_list.json / images 生成
        ↓
任务类型识别或人工指定
  - A1 法律法规
  - C1 事故报告
  - C3 隐患图集
        ↓
类型化算法处理
  - 法规结构恢复
  - 事故报告本体抽取
  - 隐患图集条目提取
        ↓
业务结果输出
  - Markdown
  - JSON
  - 图片资源
  - 目录化样本库
```

## 7. A1 法律法规结构恢复算法

### 7.1 适用范围

适用于法律法规、部门规章、地方规定、标准规范、管理办法等文档，核心任务是恢复法规天然层级。

典型结构包括：

- 章 - 节 - 条
- 章 - 条
- 仅条文的扁平法规
- 数字编号型标准规范，如 `1.0.1`、`8.6.1`

### 7.2 输入

```text
Mineru_changed/{doc_name}/auto/
├── *_content_list.json
├── *.md
└── images/
```

### 7.3 输出

```text
regulation/{doc_name}/
├── content_list.json
├── document.md
└── images/
```

### 7.4 核心流程

```text
MinerU content_list
        ↓
文本块原子化切分
        ↓
阶段 1：全局标题分析 Agent
        ↓
阶段 1 摘要压缩
        ↓
阶段 2：内容分段处理 Agent
        ↓
结果合并
        ↓
生成 content_list.json + document.md
```

### 7.5 设计要点

- 采用两阶段处理：先全局恢复文档骨架，再局部识别子标题。
- 使用人工记忆模拟机制，降低跨 chunk 层级漂移。
- 保留法规原始章、节、条关系，不把法规简单压平成普通段落。
- 默认 LLM-first，规则仅作为可选补充。

## 8. C1 事故调查报告本体抽取算法

### 8.1 适用范围

适用于事故调查报告、政府事故批复、调查组报告、事故附件材料等复杂文档，核心任务是根据专家本体抽取结构化字段。

### 8.2 输入

```text
input_file_mix/         # 原始 Word / PDF
input_file_pdf/         # 归一化后的 PDF
output_file/            # MinerU 解析结果
Dataset/                # 标题层级整理后的 Markdown 数据集
```

### 8.3 输出

```text
ontology_output_v2/
├── *_ontology.json
├── *_memory.json
└── _processing_summary.json
```

### 8.4 核心流程

```text
Step1 Word/PDF 归一化
        ↓
Step2 MinerU 文档解析
        ↓
Step3 标题层级组织
        ↓
Step4 Markdown 标题回写
        ↓
Step5 本体约束 Agent 抽取
        ↓
结构化事故报告 JSON
```

### 8.5 Agent 架构

```text
标准化 Markdown
        ↓
HeaderExtractor
        ↓
HeaderSummarizer
        ↓
SplitPlanner
        ↓
DocumentSplitter
        ↓
InformationExtractor
        ↓
OntologySerializer
        ↓
结构化事故报告 JSON
```

### 8.6 设计要点

- 专家本体定义“抽什么”和“怎么抽”，字段扩展优先修改本体。
- 抽取策略强调复制原文，避免自由生成导致事实漂移。
- 支持跨 chunk 汇总，适配事故经过、责任认定、管理缺陷等分散信息。
- 输出 `*_memory.json`，支持调试、审计和结果复盘。

## 9. C3 安全隐患图集提取算法

### 9.1 适用范围

适用于安全隐患图集、图文样本库、隐患条目资料等文档，核心任务是按隐患条目组织目录、描述字段和对应图片。

当前重点处理：

```text
4 工程实体安全隐患
```

### 9.2 输入

```text
Mineru_changed/隐患图集/{doc_name}/auto/
├── *_content_list.json
├── *.md
└── images/
```

### 9.3 输出

```text
hazard_album_output/{图集名称}/{章节}/{隐患类型}/{隐患名称}/
├── section_info.txt
├── 隐患图片1.jpg
├── 隐患图片2.jpg
└── 正确实例.jpg
```

`section_info.txt` 格式：

```text
隐患问题：...
风险分析：...
整改措施：...
规范要求：...
```

### 9.4 核心流程

```text
MinerU content_list
        ↓
章节识别与路径构建
        ↓
section 聚合
        ↓
过滤目标章节和含图 section
        ↓
字段抽取
        ↓
目录树生成 + 图片复制 + section_info.txt 写入
```

### 9.5 设计要点

- 使用规则驱动，不依赖 LLM，保证输出稳定、成本低、可复现。
- 保留“图集 > 章节 > 隐患类型 > 隐患名称”的目录树结构。
- 将图片和四类文本描述绑定为一个隐患知识单元。
- 显式处理 Windows 非法路径字符、重名图片和相对路径问题。

## 10. 统一输入输出规范

### 10.1 输入形式

统一支持以下输入：

- 原始 Word 文件：`.doc`、`.docx`
- 原始 PDF 文件：`.pdf`
- MinerU 解析结果：`content_list.json`、Markdown、`images/`

### 10.2 中间结果

```text
standardized_document/
├── document.md
├── content_list.json
└── images/
```

其中：

- `document.md` 是面向阅读和后续 Agent 处理的标准文本。
- `content_list.json` 是保留节点类型、标题层级、图片路径等信息的结构化中间结果。
- `images/` 是文档中的图片、表格截图或图集资源。

### 10.3 输出形式

| 任务类型 | 主要输出 | 输出用途 |
|---|---|---|
| A1 法律法规 | `document.md`、`content_list.json` | 法规知识库、检索、问答、RAG 切片。 |
| C1 事故报告 | `*_ontology.json`、`*_memory.json` | 事故知识图谱、结构化字段入库、责任链分析。 |
| C3 隐患图集 | 目录树、`section_info.txt`、图片 | 图文检索、隐患样本库、前端卡片展示。 |

## 11. 关键设计原则

### 11.1 Markdown 是中间态，JSON 是结构化交付态

历史对接更关注 JSON，因为 JSON 方便系统入库和接口传输。当前补充 Markdown 示例，是为了说明算法入口和处理中间态，而不是替代最终结构化结果。

### 11.2 按任务类型选择算法

法律法规、事故报告和隐患图集的结构差异很大，应按 A1/C1/C3 分别走不同算法链路，而不是使用同一套 prompt 或规则。

### 11.3 保留原文结构，减少生成改写

三类算法都以保留原文结构为优先目标：

- A1 保留法规章、节、条层级。
- C1 优先复制原文，再按本体字段归并。
- C3 保留图集目录树和图片对应关系。

### 11.4 输出可审计、可追溯

算法输出应尽量保留来源路径、章节路径、图片路径和处理日志，使结果可复核、可调试、可持续迭代。

## 12. 成功标准

统一 PRD 对应的算法体系应满足以下要求：

- 能清晰区分 A1、C1、C3 三类任务，并选择对应处理链路。
- 能说明历史对接算法以 JSON 结构化导出为主。
- 能提供明确的“文件 -> Markdown”转换示例。
- A1 能从 MinerU 输出中恢复法规真实层级，并产出 `document.md` 与 `content_list.json`。
- C1 能从事故调查报告中稳定产出本体约束的结构化 JSON。
- C3 能从隐患图集中抽取隐患条目、描述字段和图片资源。
- 输出结果可用于知识库构建、检索问答、图文样本整理和前端展示。
